# Issue Generation Protocol

How new tasks enter the Pauli-Test. This document defines the process, the model eligibility rules, and the separation of concerns that keeps task design independent of task execution.

---

## Why a Protocol?

The most serious governance problem in any benchmark is that the same party designs the tasks and evaluates performance. The Pauli-Test addresses this at two levels:

1. **Tasks come from genuine project needs** — QUASI issues are engineering requirements, not puzzles constructed to be testable.
2. **Task framing is generated by a rotating model pool** — no human selects the specific wording, scope, or focus of individual issues. A model analyzes the project state and writes the prompt; a different model (or a human contributor) executes it.

This separates the cognitive work of identifying what the project needs from the cognitive work of doing it. The models involved in generation have different blind spots from the models being evaluated. Tasks are more likely to probe real capability gaps rather than what any single model is comfortable producing.

---

## Two-Stage Generation

### Stage 1 — Analysis (the prompt-writer)

A model receives:
- The current repository state (file tree, recent commits, open issues)
- The Capability Ladder definition (L0–L4, advancement criteria)
- The label taxonomy (compiler, specification, infrastructure, agent-ux, docs, good-first-issue)
- A brief: "Identify what the project needs next at the current frontier level. Write a GitHub issue: title, description, acceptance criteria, label."

The model produces a draft issue. It does not submit the issue. It hands off the draft.

### Stage 2 — Execution (the solver)

A different model (or human contributor) receives the issue and attempts to resolve it via pull request. The solver does not know which model wrote the issue.

### Why two stages matter

If the same model writes and solves the issue, it is evaluating itself. A model that writes vague acceptance criteria can pass its own vague test. Two-stage generation forces the acceptance criteria to be meaningful to a model that didn't author them — and to the CI pipeline, which has no opinion about who wrote what.

## Model Eligibility for Issue Generation

### Required

| Criterion | Rule |
|-----------|------|
| **Open weights** | Model weights must be publicly released under any license (Apache 2.0, MIT, Llama Community License, research license, etc.). Closed-weights models are not eligible. |
| **Third-party hosted** | The model must be accessible via a third-party hosted API. QUASI does not operate inference infrastructure. Eligible providers include OpenRouter, Together.ai, Fireworks.ai, Groq, Hugging Face Inference API, Replicate, and equivalent services. |
| **Versioned model ID** | The exact model version used must be pinned (e.g. `deepseek-ai/deepseek-v3` at a specific revision). Unversioned or floating aliases are not eligible — results must be reproducible. |
| **Coding capability** | The model must demonstrate basic code comprehension at L0 level before being admitted to the rotation. A simple admittance task is defined below. |

### Not required

- The model need not be open source in the strict sense (training data, training code). Weights alone are sufficient.
- The model need not be free to use. Third-party hosting costs are the responsibility of whoever runs the generation step.
- The model need not be multilingual or general-purpose. A coding-specialist model is fine.

### Explicitly excluded

- Any model whose weights are not publicly downloadable, regardless of API availability (GPT-4o, Claude, Gemini, Grok, and equivalents are excluded as issue generators)
- Models where the version cannot be pinned (e.g. services that update silently)

## Admittance Task

Before a model enters the generation rotation, it must pass the following task without human correction:

> **Task:** Given the QUASI repository README and the Capability Ladder definition, produce one valid GitHub issue at the current frontier level. The issue must include: a title, a description (≥3 sentences), acceptance criteria (≥2 bullet points), and a label from the taxonomy.

A human reviewer checks that the issue is coherent and genuinely advances the project. This is a one-time gate, not a recurring test. Passing models are added to `docs/ELIGIBLE-MODELS.md`.

---

## Rotation Policy

- The generation pool rotates models across issues to prevent any single model's framing from dominating the task set.
- No model may write two consecutive issues at the same capability level.
- The model used for Stage 1 (analysis) is recorded in the issue's metadata field and in the quasi-ledger at the time of issue creation.
- The Stage 1 model is excluded from solving the issue it generated (Leaderboard B enforcement).

---

## Ledger Entry for Generated Issues

When an issue is created via this protocol, a ledger entry is appended:

```json
{
  "type": "issue_created",
  "task": "QUASI-NNN",
  "level": 1,
  "generator_model": "mistralai/mistral-large-2411",
  "generator_provider": "openrouter",
  "timestamp": "2026-02-24T...",
  "hash": "sha256:..."
}
```

This makes the generation provenance auditable independently of GitHub.

## Access

The generation script is at `quasi-agent/generate_issue.py`. It requires:

```bash
OPENROUTER_API_KEY=...   # or provider-specific key
QUASI_GENERATOR_MODEL=mistralai/mistral-large-2411   # pinned version
GITHUB_TOKEN=...         # to open the draft issue
```

Running the script:

```bash
python quasi-agent/generate_issue.py \
  --level 1 \
  --dry-run          # prints draft without submitting
```

Remove `--dry-run` to open a draft GitHub issue for human review before publication.

---

## Governance Note

The generation protocol is itself subject to the same open governance as the rest of QUASI. Changes to this document require a pull request and CI approval. The model rotation list in `docs/ELIGIBLE-MODELS.md` is community-editable — any contributor may nominate a model by opening an issue and demonstrating admittance task passage.

## Validation checklist

Before opening a generated issue, verify:

- the title is specific and non-duplicative
- the acceptance criteria are concrete and testable
- the proposed scope matches the selected Pauli-Test level
- the issue does not duplicate an already open workflow/badge task
